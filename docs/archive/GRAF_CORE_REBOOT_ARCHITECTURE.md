# GRAF CORE Reboot Architecture — Start = Living Graph

## SECTION 1 — PRODUCT THESIS

### What GRAF actually is
GRAF is a **personal operating system built as a living probability graph**. It is not a form, not a scorecard, and not a static analytics page. It is the place where a user continuously maps reality, receives predictions, and executes next moves.

### What job it does for the user
GRAF helps a user answer one practical question, repeatedly: **“Given my current state, what should I do next to increase my probability of desired outcomes?”**

### Why graph-first is the correct interaction model
A user’s life/system is not linear. Causes and effects are networked:
- Sleep impacts focus.
- Focus impacts output quality.
- Output quality impacts confidence and opportunities.
- Financial stress impacts sleep again.

A graph is the native structure for this reality: nodes hold state, edges hold influence, and paths hold strategy.

### Why the old Start model fails
The old Start pattern behaves like a verdict screen:
- asks for disconnected inputs,
- returns decorative certainty,
- hides the causal model,
- delays agency.

It tells users “what they are” instead of letting them **operate the system**.

### What new Start/Graph must achieve (plain language)
From minute one, the user must feel:
1. “I can start anywhere meaningful.”
2. “My input changes the system immediately.”
3. “The system explains why this matters.”
4. “I can see risk and opportunity nearby.”
5. “I have a clear best next move right now.”

## SECTION 2 — CORE CONCEPT

### Start = first mode of the graph
The first screen is a live graph canvas with one action: **Choose Your Central Node**.

### What the user sees first
- A dark, calm graph field.
- 6–8 candidate central domains as large interactive nodes (e.g., Energy, Focus, Mood, Work Throughput, Physical Resilience, Financial Stability).
- Each node has a one-line plain-language promise: “Improve this and three nearby systems stabilize.”

### How the first central node is chosen
User picks one of two methods:
1. **Direct choice** (manual control).
2. **Fast lock-on** (3 micro-prompts, <45 sec) suggesting best starting node with confidence band.

The user always confirms final selection.

### What happens after choosing it
The selected node becomes the center of a local graph cluster (“Sector 0”).
- First-ring neighboring nodes appear.
- Edge strengths appear as simple labels: weak / medium / strong influence.
- A “Stability Pulse” bar appears for the center node.

### How nearby nodes/domains appear
Unlock logic:
- immediate neighbors: visible but partially masked,
- second-ring neighbors: silhouettes,
- deeper graph: fogged map.

The user sees there is a bigger world, but only relevant local complexity is active.

### How tools are revealed inside the graph
Every node has embedded tools in-context:
- **Capture tool** (input)
- **Interpret tool** (what this means)
- **Act tool** (next intervention)

Tools open as side-panels anchored to node/edge, never as disconnected pages.

### How graph changes as data is entered
When user logs data or confirms an event:
- node state updates (color + pulse + confidence),
- connected edge weights may strengthen/weaken,
- predicted route shifts,
- risk markers update,
- “best next move” can re-rank in seconds.

The user learns: **input is power, not paperwork**.

## SECTION 3 — SYSTEM MODEL

- **Node**: A stateful unit the user can observe and improve (e.g., Sleep Regularity). It has value, trend, confidence, and active tools.
- **Edge**: A directional influence between nodes with weight and uncertainty (e.g., Sleep → Focus, strong positive, medium confidence).
- **Domain**: A cluster of related nodes serving one life function (e.g., Cognitive Performance).
- **Layer**: A depth tier of detail inside a domain (from simple operational signals to advanced diagnostics).
- **Unlock**: A rule-based reveal of nodes/tools/layers triggered by evidence, completion, time consistency, or dependency resolution.
- **Tool**: A graph-native operation that captures data, computes insight, or triggers action directly on nodes/edges.
- **Route**: A multi-step plan across nodes that targets an outcome with expected probability and effort cost.
- **Confidence**: How reliable the system’s estimate is, based on recency, completeness, consistency, and cross-signal agreement.
- **Risk**: Probability of undesirable drift/outcome if no intervention is applied.
- **Best next move**: The highest expected-value action right now, balancing impact, effort, confidence, and dependencies.

## SECTION 4 — FIRST-SESSION FLOW (FIRST 5–10 MINUTES)

1. **First impression (0:00–0:30)**
   - User enters live graph immediately.
   - System message: “Choose where to take control first.”

2. **First choice (0:30–1:30)**
   - User selects central node (manual or fast lock-on).
   - Graph recenters and expands first-ring context.

3. **First data extraction (1:30–3:30)**
   - Node opens a 90-second “Power Scan”: 4–6 ultra-targeted inputs.
   - Inputs are interaction-rich (sliders, event chips, confidence toggles), not long forms.

4. **First reward (3:30–4:00)**
   - “Signal Ignition” animation: node stability score appears with confidence band.
   - User receives first unlock token (“Path Fragment”).

5. **First graph reaction (4:00–6:00)**
   - 2–3 neighboring nodes illuminate based on scan.
   - One hidden dependency becomes visible (“Late caffeine -> Sleep latency risk”).

6. **First recommendation (6:00–8:00)**
   - System presents one immediate action + one strategic action.
   - Each action includes expected effect window and probability range.

7. **First feeling of power/control (8:00–10:00)**
   - User executes one action commitment inside node.
   - Route preview updates instantly (“If completed, focus-route confidence +12%”).
   - User exits session with a visible, owned trajectory—not a static score.

## SECTION 5 — GAMIFICATION SYSTEM

### Core loop (serious, retention-driven)
**Observe -> Input -> Unlock -> Act -> Impact -> Expand graph -> Repeat**

### Progression
- Account-level: Operator Rank (based on consistency + resolved risks).
- Domain-level: Mastery Track per domain.
- Node-level: Node Calibration Level.

### Unlocks
- New node visibility.
- Deeper tools (forecast simulators, dependency inspectors).
- Higher-confidence forecasting windows.
- Cross-domain routes.

### Levels of depth per domain
Each domain has depth gates tied to evidence quality, not only time spent.

### Rewards
- Functional rewards first (new capability, reduced uncertainty, smarter routes).
- Cosmetic rewards second (theme variants, graph signatures).

### Streaks (with alternatives)
No fragile “perfect streak” punishment.
Use:
- **Momentum meter** (decays gracefully),
- **Recovery quests** after missed days,
- **Stability bonus** for realistic consistency.

### Quests/challenges
- Daily micro-quests: 2–5 minute high-value checks.
- Weekly system quests: “Resolve one high-risk edge chain.”
- Scenario challenges: “Travel week adaptation route.”

### Visible growth of the graph
The map physically expands with meaningful progress:
- fog clears,
- edge certainty sharpens,
- route bandwidth increases.

### “You opened a deeper layer of yourself/system”
Depth unlock moments should state capability gain explicitly:
“New layer unlocked: Circadian Drift Diagnostics. You can now distinguish fatigue cause vs schedule misalignment.”

### Avoid childish cringe while staying addictive
- Visual language: premium instrumentation, not cartoon trophies.
- Reward language: “capability unlocked” not “gold star”.
- Progress framing: system control, reduced uncertainty, strategic advantage.

## SECTION 6 — INFORMATION DEPTH MODEL

Use naming that signals operational maturity:
1. **Surface** — core state read + simple actions.
2. **Pattern** — trend and trigger visibility.
3. **Mechanism** — causal diagnostics and edge-level analysis.
4. **Strategy** — route simulation and trade-off planning.
5. **Orchestration** — cross-domain optimization with long-horizon forecasting.

### How each level opens
- Surface: immediate.
- Pattern: minimum recurring data + 3-day continuity.
- Mechanism: sufficient signal diversity + validated dependencies.
- Strategy: demonstrated action completion + stable confidence.
- Orchestration: multi-domain integrity and risk management competency.

### What user gains at each level
- Surface: clarity.
- Pattern: anticipation.
- Mechanism: explanation.
- Strategy: control.
- Orchestration: compounding outcomes.

### Domain balance rules
- Every domain should expose equivalent depth architecture, even if content differs.
- No domain should become a dead-end toy.

### Can one domain outpace others?
Yes, but with tapering returns.
Advanced layers in one domain eventually require minimum competence in connected domains to preserve model validity.

### Cross-domain restrictions
Soft restrictions only:
- gate highest-risk interventions if prerequisite nodes are unknown,
- explain gating in plain language,
- always provide alternative progress paths.

## SECTION 7 — UX / UI PRINCIPLES

- The graph should feel like a **live control room**, not an infographic.
- Always visible: central node health, top risks, best next move, confidence indicator.
- Hidden until needed: advanced diagnostics, deep technical traces, low-relevance domains.
- Prevent overload via progressive disclosure and locality (focus radius around active node).
- Keep readability with edge hierarchy (only strongest/relevant edges fully rendered).
- Make reactivity explicit: every user action triggers visible node/edge/system response.
- Avoid decorative traps: no animation without state meaning.
- Every visual element must answer one of three jobs: **state, causality, action**.

## SECTION 8 — RELATIONSHIP TO DATA SUBSTATION

### What belongs in GRAF CORE
- User-facing operating graph.
- Action recommendations.
- Plain-language confidence/risk.
- Node/edge interaction tools.

### What belongs in Data Substation
- System-wide telemetry and diagnostics.
- Data quality audits.
- Dependency break detection.
- Model drift, stale signals, inactive function monitoring.
- Calibration controls and inspection layers.

### What should never be mixed
- Raw technical diagnostic noise inside everyday CORE flow.
- Gamified UX mechanics inside Substation diagnostic layers.

### How they connect
- CORE consumes interpreted outputs from Substation (health flags, confidence updates, anomaly warnings).
- Substation observes CORE interactions and model performance continuously.
- Shared state contract: node health, edge confidence, data freshness, forecast validity.

### Day-one Substation states to monitor
1. Node data completeness.
2. Edge confidence stability.
3. Stale critical variables.
4. Broken dependency chains.
5. Forecast error trend.
6. Inactive but required tools.
7. User action-to-outcome lag anomalies.

## SECTION 9 — MVP CUT

### Must exist in v1
- Start = live graph entry.
- Central node selection.
- One fully operational domain + two connected neighbor domains.
- Node tools: Capture / Interpret / Act.
- Basic confidence + risk display.
- Best next move engine (simple expected-value model).
- First unlock loop (visible graph expansion).

### Postpone
- Full multi-domain orchestration.
- Complex long-horizon simulations.
- Heavy personalization aesthetics.
- Social/community mechanics.
- Advanced Substation controls exposed to end users.

### Fake complexity to cut
- Overly detailed score taxonomies with no action consequence.
- Fancy but uninformative graph animation.
- Massive questionnaires pretending to be intelligence.
- Premature “AI coach chat” replacing explicit causality.

### First real “holy sh*t this is powerful” moment
After 3–5 minutes of input, the graph reveals a non-obvious dependency chain, quantifies risk, and offers one actionable move that the user immediately executes—then the route probability visibly improves in real time.

## SECTION 10 — OUTPUT FORMAT

### A) One-sentence product definition
GRAF is a living personal operating graph that turns user input into probabilistic next moves across connected life domains.

### B) One-paragraph concept summary
GRAF starts as an interactive graph, not a questionnaire: the user chooses a central node, enters high-value signals directly in that node, and watches nearby domains, risks, and opportunities update live. Every interaction modifies node health, edge confidence, and route forecasts, producing a clear “best next move” with transparent probability. Gamification is used as capability progression—unlocking deeper layers, stronger tools, and broader graph territory—so engagement translates into better data and better decisions. Data Substation runs in parallel as a serious monitoring and calibration system that powers reliability without polluting the user-facing CORE experience.

### C) 10 most important design laws for new GRAF CORE
1. Start is always the graph.
2. Every input must change state visibly.
3. Expose causality, not just conclusions.
4. Recommend actions, not labels.
5. Show confidence with every important claim.
6. Gate depth by evidence quality, not arbitrary time.
7. Prioritize local relevance over global complexity.
8. Reward capability unlocks over cosmetic dopamine.
9. Keep CORE human-readable and Substation technically rigorous.
10. If a visual element has no decision consequence, remove it.

### D) Do not build this wrong again (anti-patterns)
- A form-first funnel that delays graph interaction.
- Static scores with no actionable route.
- Decorative network graphics disconnected from logic.
- “One number explains everything” oversimplification.
- Hidden confidence/uncertainty.
- Endless onboarding before value appears.
- Punitive streak mechanics causing churn.
- Mixing diagnostic engineering screens into everyday user flow.
- Unlocks that add noise instead of capability.
- Replacing model transparency with vague motivational copy.
